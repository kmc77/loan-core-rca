Apache Fineract · Root Cause Analysis

Loan Re-aging의 실재했던 결함 재현과 RCA

Apache Fineract 1.10–1.12에 실제로 출시됐던 re-age 결함(FINERACT-2384)을, 해당 결함이 그대로 들어있는 base 버전에서 재현하고 1.13.0의 공식 수정 전략으로 해소하는 과정을 기록한다.

티켓 FINERACT-2384 (Bug) 영향 버전 1.10.1 / 1.11.0 / 1.12.1 수정 1.13.0 기반 apache/fineract fork
00

정정 고지

이 저장소의 초기 서술은 사실이 아니었다. 명확히 바로잡는다.

철회된 주장

초기 README는 “Re-aging이 승인원금을 참조하는 버그를 발견·수정했다”고 주장했다. 그런 버그는 Fineract에 존재한 적이 없다. FINERACT-1971은 버그 티켓이 아니라 re-aging 최초 구현 기능 티켓이고, 그 코드는 base 커밋부터 잔여원금 기준으로 정상 동작한다. 해당 “버그”는 본 저장소가 임의로 주입했다가 원복한 합성 시나리오였다.

아래는 그 가짜 서사를 폐기하고, 실제로 출시됐던 결함을 그대로 재현하는 것으로 프로젝트를 재정의한 것이다. 정직성 확보가 신뢰의 전제다.

01

실제 결함: FINERACT-2384

Apache JIRA에서 확인된 실제 Bug 티켓이다. Reporter/Assignee는 Fineract 리드(Adam Saghy), 영향 버전 1.10.1 / 1.11.0 / 1.12.1, 수정 버전 1.13.0.

결함 내용

  • 트랜잭션 날짜 오류: re-age 트랜잭션 날짜를 항상 “현재 영업일”로 고정한다. 올바른 동작은 — re-age 시작일이 현재 영업일보다 이르면 트랜잭션 날짜는 re-age 시작일, 그렇지 않으면 현재 영업일.
  • 만기일 엣지케이스: re-age 시작일이 대출 만기일과 동일하면, 시작일·만료일이 모두 만기일이고 일수가 0인 잘못된 신규 회차가 스케줄에 생성된다.

출처: Apache JIRA — issues.apache.org/jira/browse/FINERACT-2384 · 관련: FINERACT-2385 (0원 re-age 트랜잭션 금지).

02

이 결함은 이미 base 코드에 있다

본 저장소의 base 커밋은 4d4502fbe — FINERACT-1971: Loan re-aging foundational implementation. 이는 2384 수정(1.13.0) 이전의 버그 버전이다. 따라서 결함을 주입할 필요가 없다. LoanReAgingServiceImpl.createReAgeTransaction(...)에 원본 그대로 존재한다:

private LoanTransaction createReAgeTransaction(Loan loan, JsonCommand command) {
    ExternalId txExternalId = externalIdFactory.createFromCommand(command, ...);
    LocalDate transactionDate = DateUtils.getBusinessLocalDate();  // ← FINERACT-2384 그 자체
    ...
}
검증됨

transactionDate를 무조건 현재 영업일로 하드코딩한 이 한 줄이 곧 2384의 트랜잭션 날짜 결함이다. 본 저장소의 기존 원금 작업(주입/원복)은 이 줄을 건드린 적이 없다 — 진짜 결함은 손대지 않은 원본 그대로 코드에 살아있다.

03

재현 시나리오

가짜 버전은 금액(1250 vs 1000)을 검증했다. 실제 2384는 트랜잭션 날짜와 스케줄 구조를 검증한다.

시나리오 A — 트랜잭션 날짜 오류

#행위값
1대출 실행(Disbursement)2023-02-01
2영업일을 뒤로 진행→ 2023-06-01
3re-age 실행 (start date = 과거)start 2023-02-02
  • 기대(정상/2384 수정 후): re-age 트랜잭션 날짜 = 2023-02-02 (start date).
  • 실제(버그 base): re-age 트랜잭션 날짜 = 2023-06-01 (현재 영업일). → 재현.

시나리오 B — 만기일 엣지케이스

  • re-age 시작일을 대출 만기일과 동일하게 설정.
  • 기대: 정상 스케줄.
  • 실제(버그): 시작일=만기일·만료일=만기일·일수 0인 쓰레기 회차가 스케줄에 추가됨. → 재현.

통합테스트 단언 포인트는 트랜잭션 객체의 날짜 필드, 그리고 상환 스케줄에 0일짜리 회차가 존재하는지 여부다.

04

근본 원인

re-age는 “과거 시점부터의 워크아웃 약정”을 표현할 수 있어야 한다. 트랜잭션 날짜를 무조건 처리 시점(현재 영업일)으로 고정하면, 소급 적용된 약정이 원장에 잘못된 일자로 기록되어 연체 계산·스케줄 재산정의 기준 시점이 어긋난다. 만기일 엣지케이스는 경계 조건(시작일 == 만기일)에 대한 가드 부재로, 길이 0인 무의미한 회차를 생성한다. 둘 다 단순 표시 문제가 아니라 스케줄/원장 정합성 결함이다.

05

실제 수정 전략

1.13.0이 채택한 문서화된 전략을 그대로 구현한다.

LocalDate reAgeStartDate = command.dateValueOfParameterNamed(LoanReAgingApiConstants.startDate);
LocalDate businessDate   = DateUtils.getBusinessLocalDate();

// 시작일이 영업일보다 이르면 시작일, 아니면 영업일
LocalDate transactionDate = DateUtils.isBefore(reAgeStartDate, businessDate)
        ? reAgeStartDate
        : businessDate;

추가로 reAgeStartDate가 만기일과 같을 때 0일 회차가 생성되지 않도록 스케줄 생성부에 가드를 둔다. 정확한 적용 위치는 실제 코드 RCA 과정에서 확정한다 — 그것이 이 프로젝트의 본 목적이다.

06

프로젝트 상태와 범위

검증 완료

FINERACT-2384가 실제 Bug 티켓이며 영향 버전·수정 버전이 확인됨(JIRA). 결함 코드가 본 base 커밋에 원본 그대로 존재함(소스 직접 확인).

진행 예정

시나리오 A/B 통합테스트 작성, 버그 base에서 실패 재현, 수정 전략 적용 후 그린 확인. 통합테스트는 실행 환경(DB 등)을 요구하며, 그린 여부는 로컬 재현을 전제로 한다 — 본 문서는 검증되지 않은 “통과”를 단정하지 않는다.

단일 시나리오 연습이다. 본 프로젝트의 가치는 산출물 자체가 아니라 상태 의존 코어뱅킹 결함을 해당 버전에서 결정적으로 재현하고 공식 수정 전략으로 해소하는 RCA 절차의 훈련에 있다.

07

커밋 이력 메모

기존 저장소에는 폐기된 가짜 서사의 흔적이 남아 있다. 정리 대상이다.

b968deb10 test & bug: 가짜 결함 주입 + 테스트 본 저장소 — 폐기 대상
5f9845f86 FINERACT-1971: loan level event 트리거 Apache 원본
4d4502fbe FINERACT-1971: re-aging foundational impl. (base · 2384 결함 포함) Apache 원본

가짜 원금 작업은 base 상태로 되돌리고, 그 위에서 시나리오 A/B 재현 → 2384 RCA → 공식 수정 순으로 다시 쌓는다. 커밋 메시지는 “주입”이 아니라 “FINERACT-2384 재현/수정”으로 일관되게.

본 저장소는 Apache Fineract의 fork이며 Apache-2.0 라이선스를 따른다. 원본 소스·라이선스·NOTICE는 상위 프로젝트를 따른다.
